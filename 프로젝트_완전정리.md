# Part 1. 개념 토대 (블록체인 기초부터)

## 1.1 이 프로젝트 한 줄 정의
증권사(계좌관리기관) 자리에서 토큰증권을 **발행·청약·배정·권리이전**하고, 그 결과를
**온체인 정본(전자등록계좌부)과 오프체인 권리자명부 DB가 어긋나지 않게 유지**하는 백엔드 + 스마트컨트랙트.

## 1.2 배경 (왜 지금)
2026년 1월 전자증권법·자본시장법 개정으로 **분산원장이 법적 전자등록계좌부로 인정**되고
(시행 예정 2027년 1월), 증권사가 계좌관리기관/유통 인프라를 직접 운영하게 된다.
이 프로젝트는 그 구조를 로컬 PC 한 대로 축소 재현한 것이다. 평가 포인트는 화려함이 아니라
**① 온/오프체인 정합성 ② 전송 제약의 코드 강제 ③ 장애 처리**.

## 1.3 등장인물 (도메인 → 구현)
| 주체 | 현실 | 이 프로젝트 |
|---|---|---|
| 발행인 | 증권을 찍는 회사 | mock(REST 요청 시뮬레이션) |
| 전자등록기관 | 한국예탁결제원(KSD) 등 | mock(자동 승인) |
| **계좌관리기관(증권사)** | 투자자 계좌·장부 운영 | **← 이 백엔드 (서명 계정 = agent)** |
| 투자자 | 증권을 사는 사람 | REST 클라이언트 |

## 1.4 블록체인이 뭔지 — 노드와 장부
- **블록체인 = 여러 컴퓨터가 함께 들고 똑같이 유지하는 "공유 장부"**. 한번 기록되면 위변조가 어렵다.
- **노드(node) = 그 장부를 들고 운영하는 한 대의 참여자(은행원)**. 노드가 4개라도 **장부는 1개**, 4명이 같은 장부의 복제본을 들고 항상 일치시킨다.
- **노드 프로그램** = 노드를 실행하는 소프트웨어. 우리는 둘을 쓴다:
  - **Anvil**: 개발용 **단일 노드**(혼자, 즉시 처리). 빠른 개발/테스트용.
  - **Hyperledger Besu**: 실무급 노드 프로그램. **여러 노드가 합의**(QBFT)할 수 있음 → "여러 기관 공동 운영" 시연용.
  - 둘은 같은 역할(블록체인 노드)의 다른 프로그램. 전파·합의 기능은 프로그램에 **내장**돼 있다(우리가 코드로 안 만든다).

> 비유: 블록체인 = 모두가 동기화해 보는 구글 문서 1개. 노드 = 그 문서를 띄워둔 사람. 누가 고치면 전원에게 똑같이 반영.

## 1.5 두 개의 장부 — 이 프로젝트의 본질
"권리자명부(누가 얼마 보유)"가 **두 벌** 존재한다.

| | 어디에 | 역할 |
|---|---|---|
| **온체인 장부** | 블록체인 컨트랙트의 `balanceOf` | **정본(법적 진짜)** |
| **오프체인 장부** | PostgreSQL `holding` 테이블 | **미러(빠른 조회용 사본)** |

왜 둘? 블록체인은 위변조 불가·법적 정본이지만 "이름으로 검색", "종목별 보유자 목록"이 느리고 불편하다.
→ **정본은 온체인, 조회는 DB 사본**으로 나누고, **둘을 일치시키는 것**이 이 프로젝트의 핵심.

## 1.6 RPC = 노드에 거는 "전화번호"
백엔드가 블록체인에 말을 걸려면 노드가 요청을 받는 네트워크 주소가 필요하다. 그게 **RPC 주소**.
- 예: `http://127.0.0.1:8545` (127.0.0.1=내 PC, 8545=노드가 듣는 포트).
- **어느 블록체인을 쓸지는 이 RPC 주소로 결정**된다. Anvil↔Besu 전환 = RPC 주소만 바꾸면 끝(아래 Part 5).

## 1.7 컨트랙트 = 장부 위의 "프로그램"
- **스마트컨트랙트**: 블록체인 안에 배포되어 돌아가는 프로그램. 우리 것은 `SecurityToken.sol`.
- **배포(deploy)**: 컨트랙트를 한 노드에 보내면 → 합의를 거쳐 **공유 장부에 한 번 기록** → 모든 노드가 **같은 주소·같은 상태**로 복제. (노드마다 다른 게 아니다)
- **호출 두 종류**:
  - **조회(call)**: `balanceOf` 같은 읽기. 상태 안 바뀜, 가스·서명 불필요.
  - **트랜잭션(transaction)**: `mint`/`transfer` 같은 쓰기. 상태 바뀜 → **개인키 서명 필요**, 가스 소모.
- **web3j**: 자바(백엔드)에서 위 호출을 만들어 RPC로 보내주는 라이브러리.
- **nonce**: 계정마다 매기는 거래 순번(0,1,2…). 한 거래가 블록에 들어가면 다음 번호로 올라가, **같은 거래가 두 번 적용되는 것을 원천 차단**(Part 5에서 다시).

```
[투자자/발행인 (REST 클라이언트)]

        ▼

│  계좌관리기관 백엔드 (Spring Boot)                 │

┌────────────────────────┐   ┌────────────────────────┐
│ 블록체인 (Anvil/Besu)     │   │ PostgreSQL              │
│ = 전자등록계좌부 (정본)    │   │ 권리자명부 (미러)         │

│  - whitelist/lockup      │   │ holding/transaction     │

│  - emit 이벤트            │                │
└──────────┬─────────────┘                  │
           └── 이벤트 ──► SyncService ────────┘ (온체인→DB 단방향)
```

## 2.1 온체인/오프체인 책임 분담
| 책임 | 온체인 | 오프체인(DB) |
|---|---|---|
| 권리의 정본(잔고) | ✅ 정본 | 미러(조회용) |
| 전송 제약 최종 강제 | ✅ `require` | 사전검증만 |
| 투자자 실명·KYC | 주소(식별자)만 | ✅ 실데이터 |
| 빠른 조회·집계·명부 | ✕ | ✅ |

## 2.2 컴플라이언스 이중화 (핵심 원칙)
모든 쓰기 API는 **백엔드 사전검증(KYC·whitelist·락업·한도·잔고)**으로 빠르게 실패시키고,
**컨트랙트 `require`**로 최종 강제한다. 백엔드를 우회해 컨트랙트를 직접 호출해도 온체인에서 막힌다.
→ "좋은 에러 UX + 우회 불가 신뢰"를 동시에 달성.

# Part 3. 코드 투어 (어디서부터 무엇을 보나)

⭐ = 이 프로젝트의 진짜 핵심(동기화·대사). 시간 없으면 Step 6,7 + Part 4만 봐도 된다.

## Step 1 ─ 정본 장부: [contracts/src/SecurityToken.sol](contracts/src/SecurityToken.sol)
권리의 정본이자 전송 제약이 코드로 강제되는 곳. 핵심 두 함수:
```solidity
function mint(address to, uint256 amount) external onlyAgent {        // 발행
    require(whitelisted[to], "ST: receiver not whitelisted");         // KYC 필수
    require(totalSupply + amount <= maxSupply, "ST: exceeds maxSupply"); // 한도
    ...
    emit Issued(to, amount);                                          // 이벤트
}
function _enforceTransfer(address from, address to, uint256 amount) internal { // 이전 제약
    require(whitelisted[from] && whitelisted[to], ...);               // 양측 KYC
    require(block.timestamp >= lockupUntil[from], "...locked up");     // 락업
    require(balanceOf[from] >= amount, "...insufficient balance");
    ...
    emit Transferred(from, to, amount);
}
```
- `require`가 **최종 방어선**. `transfer`(본인 서명) / `agentTransfer`(계좌관리기관 대행, `onlyAgent`) 분리.
- 모든 상태변경은 **이벤트 emit** → 이게 Step 6 동기화의 입력.

## Step 2 ─ 규칙을 테스트로 이해
[contracts/test/SecurityToken.t.sol](contracts/test/SecurityToken.t.sol) 20개. 락업 경계, 발행한도 경계,
whitelist 미통과 거부, 대행 이전 동일 제약 등. `forge test`로 즉시 실행.

## Step 3 ─ 백엔드↔체인 다리: [ChainService.java](backend/src/main/java/com/sto/platform/chain/ChainService.java)
주소를 받아 컨트랙트 래퍼를 동적 load. 쓰기(`mint`,`agentTransfer`,`setWhitelist`,`setLockup`) / 읽기
(`balanceOf`,`totalSupply`,`maxSupply`,`isWhitelisted`,`lockupUntil`) / `agentAddress()`=창고 주소.
서명 키는 실제 서명 시점에 **지연 로딩**(키 없어도 구동·조회 가능). 키는 환경변수 주입.
> 래퍼 `SecurityToken.java`는 web3j-gradle-plugin이 `contracts/src`를 컴파일해 **빌드 시 자동 생성**(미커밋).

## Step 4 ─ 사본 장부: [domain/](backend/src/main/java/com/sto/platform/domain/)
`Investor`(지갑 UNIQUE,KYC) · `Security`(컨트랙트주소,supply) · `Holding`(투자자×증권 잔고=balanceOf 미러)
· `TransactionRecord`(`tx_hash` **UNIQUE**=멱등성) · `SyncCheckpoint`(마지막 처리 블록=복구) · `Subscription`(청약).
온체인 `uint256` → DB `BigInteger`(NUMERIC).

## Step 5 ─ 업무 API: [service/](backend/src/main/java/com/sto/platform/service/) · [controller/](backend/src/main/java/com/sto/platform/controller/)
패턴 동일: **사전검증 → 온체인 호출**. 발행 예:
```java
if (investor.kyc != APPROVED) throw ...;          // 사전검증
if (!chainService.isWhitelisted(addr, wallet)) throw ...;

```

# Part 4. 심층 — 동기화·정합성 (이 프로젝트의 핵심)

- **단방향**: 쓰기 API는 온체인만 변경. DB는 이벤트로 따라온다. 양방향은 충돌·복잡도만 키움.
- **폴링(웹소켓 아님)**: 복구 모델(checkpoint 재조회)과 같은 메커니즘 → 멱등·복구·테스트에 유리. 짧은 주기로 사실상 실시간.
- **멱등성 두 층**:
  - **온체인 1회 실행** = `nonce`(블록체인이 보장).
  - **오프체인(DB) 1회 반영** = `tx_hash` UNIQUE + `existsByTxHash`(우리가 보장). 겹쳐 재조회해도 중복 반영 안 됨.
- **누락 복구**: 리스너가 죽었다 살아나도 `sync_checkpoint`부터 재조회 → at-least-once 수신 + 멱등 = **exactly-once 효과**.
- **대사·복구**: 동기화가 완벽해도 만일을 대비. 어긋나면 정본(온체인)으로 DB 보정.

# Part 5. 멀티노드 합의 (Besu QBFT)

## 5.1 Anvil vs Besu, 전환 방법
- 개발/테스트 = Anvil(단일 노드, 빠름). "여러 기관 합의" 시연 = Besu QBFT 검증자 4개.
- **전환은 RPC 주소만**: `STO_CHAIN_RPC_URL`을 Anvil(8545)→Besu(8545)로. **백엔드 코드·컨트랙트 불변**.
  (둘 다 호스트 8545를 쓰므로 실제론 "그 순간 어떤 노드를 띄웠나"로 갈린다)
- Besu는 **Berlin 포크**(EIP-1559 없음=legacy 가스). 그래서:
  - 컨트랙트 컴파일 타깃을 `evm_version="berlin"`([contracts/foundry.toml](contracts/foundry.toml))으로 맞춤(기본 shanghai의 PUSH0 회피). Anvil도 호환.
  - forge 배포 시 `--legacy`, web3j도 legacy 거래라 호환.

## 5.2 전파·합의는 어디서? → 노드(Besu) 안에서, 노드끼리 P2P로
Besu 프로그램에 P2P 전파와 QBFT 합의가 **내장**돼 있다. 백엔드·컨트랙트와 무관한 인프라 계층.
4개 띄우면 **자동**으로 굴러가되, 세 가지 설정이 맞아야 한다:
1. **같은 genesis**(장부·규칙·검증자 명단 공유) — `besu/networkFiles/genesis.json`
2. **서로 연결**(peering) — enode 부트노드 + 고정 IP(`docker-compose.besu.yml`). ← 이게 안 맞으면 peers=0, 합의 멈춤
3. **QBFT 설정**(블록주기 2초 등) — genesis 안 `qbft`

## 5.3 QBFT가 블록을 정하는 법 (정족수 = 4명 중 3명)
n=4 → f=1 견딤, 정족수 2f+1=**3표**. 블록 높이마다 **제안자 1명**(라운드로빈).
```
① 제안(Propose): 그 차례 제안자가 대기 거래를 모아 후보 블록을 방송
② 준비(Prepare): 각 검증자가 "봤고 유효함" 방송 → 3표 모이면 준비됨
③ 확정(Commit) : 각자 서명 첨부 방송 → 3표 모이면 블록 영구 확정 ✅ (서명이 블록에 박힘)
```
- **즉시 finality**: 확정되면 번복(reorg) 없음. → 동기화가 reorg 걱정 없이 바로 DB 반영 가능.
- **제안자 응답 없으면**: 타임아웃 → 라운드 체인지(새 제안자 재시도). (처음 peers=0일 때 "Moved to round 2,3…"가 그거였다)
- **비잔틴 내성**: 3표를 요구하니 악의적 1명이 가짜 블록을 못 밀고, 1명이 죽어도 3명으로 진행. → "서로 100% 안 믿어도 공동 운영" 가능.

## 5.4 같은 거래가 두 블록에? → 안 됨
- QBFT는 높이마다 **제안자 1명**이라 경쟁 블록(포크)이 없다(PoW와 다름).
- 설사 떠다녀도 **nonce**가 막는다: 거래가 한 블록에 들어가면 계정 nonce가 올라가, 같은 nonce 거래는 무효 → 중복 적용 불가.

## 5.5 검증된 사실
- 검증자 4개 + 피어 연결, 블록 2초 주기 생성, **제안자가 4노드 회전** 확인.
- **web3j 백엔드가 Besu에서 mint/transfer 처리**(balanceOf 반영) 확인.
- 주의: "블록 공동 생성" ≠ "발행 권한 공유". 발행은 여전히 단일 `agent`(`onlyAgent`). 권한 분산은 컨트랙트 권한 모델(멀티 agent/멀티시그)을 따로 바꿔야 함.

자세한 시연 절차: [besu/README.md](besu/README.md).

# Part 10. FAQ (개념 + 면접 Q&A)

**Q. "장부 = 블록체인"인가?** 블록체인은 장부를 담는 기반(그릇), 실제 장부 내용은 그 위 컨트랙트의 `balanceOf`. 정본은 온체인, DB는 사본.

**Q. RPC 주소가 뭐고, 블록체인 선택은 어디서?** RPC = 노드에 요청 보내는 전화번호. **체인 선택 = RPC 주소**(`--rpc-url`/`STO_CHAIN_RPC_URL`). `foundry.toml`은 체인 선택이 아니라 **컴파일(번역) 설정**.

**Q. 노드 먼저 뜨고, 배포하면 각 노드에 컨트랙트가 전달?** 노드는 먼저 뜬다. 배포는 "각 노드에 따로 전달"이 아니라 **하나의 공유 장부에 1번 기록 → 모든 노드가 같은 주소·상태로 복제**. 백엔드는 노드 하나의 RPC만 잡으면 됨.

**Q. Besu가 뭐야?** 블록체인 노드를 돌리는 프로그램(전파·합의 내장). Anvil(단일·개발용)과 같은 역할의 실무급 버전(멀티노드 합의 가능).

**Q. 전파·합의는 어디서?** 노드(Besu) 안에서 노드끼리 P2P로. 4개 띄우고 genesis 통일+연결+QBFT 설정이 맞으면 자동.

**Q. 같은 거래가 여러 블록에?** QBFT는 높이마다 제안자 1명이라 경쟁 블록 없음 + nonce가 중복 적용 차단. "먼저 들어간 블록에 적용, 그 뒤엔 불가".

**Q. 왜 transfer가 아니라 agentTransfer?** 백엔드는 투자자 키가 없고 agent로 서명하므로 계좌관리기관 대행. 제약은 동일.

**Q. 왜 분산원장인가/Kafka·Redis는 왜 뺐나?** 변경불가·공동관리 vs 권한형 신뢰주체(트레이드오프) / 블록체인이 곧 로그·단일 인스턴스 → 과설계 회피.

**Q. 전송 제약을 왜 컨트랙트에까지?** 백엔드 우회 방지. 코드로 강제되는 신뢰(이중화).

> **한 문장 요약**: 노드들은 하나의 장부를 공유하고(필요하면 QBFT로 여러 기관이 합의), 컨트랙트는 그 장부에 한 번 기록되어 전 노드에 복제된다. 백엔드는 노드 하나의 RPC로 거래/조회하며, 쓰기는 온체인에만 하고 DB 사본은 이벤트로 단방향 동기화 + 주기 대사로 정본과 일치시킨다. 전송 제약은 백엔드 사전검증과 컨트랙트 `require`로 이중 강제된다.
